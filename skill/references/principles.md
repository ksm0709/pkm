# PKM Principles & Know-how

이 문서는 PKM 작업을 하면서 발견한 원칙, 패턴, 노하우를 축적하는 곳이다.
새로운 인사이트가 생길 때마다 업데이트한다.

## Core Principles

### 1. 원자성 (Atomicity)
하나의 노트 = 하나의 주제. 노트가 "그리고" 로 연결되는 두 개의 주제를 다루고 있다면, 분리해야 한다.
단, 하나의 주제에 대한 여러 측면(예: 정의, 장단점, 사용 사례)은 하나의 노트에 둔다.

### 2. 연결 (Connection)
모든 노트는 최소 하나의 다른 노트와 `[[wikilink]]`로 연결되어야 한다.
고립된 노트는 발견되지 않는 죽은 지식이다.

**연결의 종류:**
- **출처 연결**: `[[2026-04-05]]` — 이 지식을 처음 기록한 데일리 노트
- **개념 연결**: `[[related-concept]]` — 관련 개념
- **상위/하위**: 태그로 관리하되, 필요시 노트 본문에서 관계 명시

### 3. 자기 언어 (Own Words)
복사-붙여넣기는 지식이 아니다. 핵심을 이해하고 자기 언어로 간결하게 서술한다.
"이해 → 요약 → 서술"의 과정을 거쳐야 기억에 남는다.

### 4. 플랫 구조 + 태그 분류
`notes/` 내에 폴더를 만들지 않는다. 태그로 분류한다.
폴더는 물리적 경계를 만들어 연결을 방해한다. 태그는 다중 분류를 허용한다.

### 5. 점진적 정교화 (Progressive Refinement)
첫 번째 버전이 완벽할 필요 없다. 데일리에 거칠게 기록하고, 반복될 때 원자 노트로 승격하고,
참조할 때마다 조금씩 다듬는다.

## Tagging Strategy

### 태그 규칙
- 영문 태그: kebab-case (`global-planning`)
- 한글 태그: 띄어쓰기 없이 (`자기계발`)
- 새 태그 생성 전 기존 태그 확인 (`pkm tags`)
- 용도별 태그 패턴 예시:
  - 사람: 이름 소문자 (`sunny`, `junho`)
  - 기술: 주제 kebab-case (`navigation`, `global-planning`)
  - 유형: 활동 타입 (`1on1`, `team`, `knowledge`)
  - 카테고리: 넓은 분류 (`학습`, `기술`, `철학`)

## Know-how

_여기에 PKM 작업 중 발견한 실용적 노하우를 추가한다._

### 승격 타이밍
- 데일리에서 같은 주제가 3회 이상 등장하면 승격 후보
- "나중에 찾아봐야지" 싶은 내용은 즉시 승격

### 좋은 노트 제목
- 주장이나 결론을 제목으로: "PostgreSQL MVCC는 스냅샷 격리를 구현한다" > "PostgreSQL MVCC"
- 검색 가능성을 고려: 나중에 어떤 키워드로 찾을까?

### Frontmatter 관리
- `id`는 파일명과 동일하게 유지 (확장자 제외)
- `aliases`는 자주 쓰는 약어나 대체 이름
- `source`는 출처 데일리 노트 날짜 (선택사항)
